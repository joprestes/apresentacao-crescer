Da Automação à Qualidade Contínua

Git, GitHub Actions e SonarQube para QAs
Autores: Joelma Prestes Ferreira & Marcelo Feiteiro
Projeto: Crescer – Qualidade de Software
A qualidade não é uma etapa. É um ciclo.
E o QA moderno precisa enxergar, automatizar e medir esse ciclo do começo ao fim.

O ciclo da qualidade

O QA deixou de ser o último elo da cadeia.
Hoje participa do ciclo inteiro:
Versionamento → Automação → Análise Contínua → Feedback → Melhoria

Ferramentas centrais e seus papéis:

🔧 Git

Controle e rastreabilidade
• Histórico completo de mudanças
• Colaboração segura entre equipes
• Branching strategies para organização
• Rollback seguro quando necessário

⚡ GitHub Actions

Automação e integração
• CI/CD automatizado
• Execução de testes em paralelo
• Deploy automático por ambiente
• Notificações e relatórios em tempo real

📊 SonarQube

Qualidade e melhoria contínua
• Análise estática de código
• Detecção de vulnerabilidades
• Métricas de cobertura de testes
• Quality Gates para aprovação

Resultado: Qualidade integrada desde o primeiro commit até a produção.
Essas três ferramentas conectam pessoas, código e cultura de qualidade.

Módulo 1

Git para QAs — Controle e Colaboração

O Git é o ponto de partida da qualidade contínua.
Cada commit conta uma história, e o QA precisa dominá-la.

Benefício principal: rastreabilidade e segurança no histórico de automação e testes.

Git: Por que dominar

Git é rastreabilidade, não apenas versionamento.

  • Cada commit revela intenção e contexto das mudanças
  • Controlar código de testes é garantir histórico e confiabilidade
  • Facilita debugging ao identificar quando bugs foram introduzidos
  • Permite rollback seguro de funcionalidades problemáticas
  • Documenta evolução dos testes automatizados
  • Habilita code review eficiente entre QAs e desenvolvedores

Impacto na qualidade:

📈 Rastreabilidade

Cada mudança tem autor, data e justificativa

🔄 Reversibilidade

Desfazer mudanças sem perder trabalho

🤝 Colaboração

Múltiplos QAs trabalhando simultaneamente

Exemplo de commits:

❌ fix tests → genérico e sem contexto
✅ test: adiciona validação de login com @smoke → claro e auditável
✅ fix: corrige timeout em teste de API de pagamento → específico
✅ feat: implementa teste E2E para fluxo de checkout → descritivo

Branches e convenções

Organização e fluxo previsível garantem rastreabilidade.

Padrões recomendados:

feature/
nova funcionalidade
fix/
correção de bug
test/
automações de teste
docs/
alterações em documentação
release/
versão candidata à entrega

Branches de release

Branches de release representam o código pronto para validação e entrega.

Servem para:

  • Congelar o código que será entregue (feature freeze)
  • Aplicar apenas correções críticas (hotfixes)
  • Testar integração e regressão sem afetar desenvolvimento ativo
  • Permitir que desenvolvedores continuem trabalhando em novas features
  • Facilitar deploy gradual (staging → produção)
  • Manter histórico de versões entregues

Fluxo de trabalho:

Criar release branch
Testes de regressão
Correções críticas
Deploy produção
Merge para main

Exemplos práticos:

release/1.5.0 → versão candidata da release 1.5
release/hml-2025-10 → branch temporária para homologação
release/sprint-42 → entrega do sprint 42
hotfix/critical-security-fix → correção urgente de segurança
Benefício: Previsibilidade no pipeline, clareza de estágios e deploy seguro.
O QA tem controle total sobre o que será testado e entregue.

Commits semânticos e Pull Requests

Formato padrão: <tipo>: <descrição>

Tipos comuns:

feat fix test refactor docs

Boas práticas:

  • Descrições curtas e específicas
  • Evitar commits genéricos
  • Um Pull Request bem escrito facilita revisão e colaboração
O commit é a menor unidade de confiança.

Módulo 2

GitHub Actions — Automação e Entrega Contínua

Automatizar é reduzir o tempo entre o commit e o feedback.
O QA participa ativamente na criação e execução desses fluxos.

Estrutura do Workflow

O arquivo .yml define o pipeline de CI/CD completo.

Principais blocos e configurações:

on: (Triggers)

Quando o fluxo é disparado
• push, pull_request
• schedule (cron jobs)
• workflow_dispatch (manual)
• release, issues

jobs: (Trabalhos)

O que será executado
• Execução paralela ou sequencial
• Diferentes sistemas operacionais
• Dependências entre jobs
• Estratégias de matriz

steps: (Passos)

Sequência de comandos
• Actions do marketplace
• Comandos shell customizados
• Condicionais (if:)
• Variáveis de ambiente

env: (Ambiente)

Variáveis e configurações
• Secrets seguros
• Configurações por ambiente
• Tokens de acesso
• URLs de APIs

Fluxo típico de QA:

Checkout código
Setup ambiente
Instalar dependências
Executar testes
Análise SonarQube
Gerar relatórios
Deploy/Publicar
Benefício: Automação completa do ciclo de qualidade, desde o commit até a entrega.

Execução por tags

Permite rodar testes específicos, otimizando tempo de execução e recursos.

Estratégias de tagueamento:

@smoke @regression @api @e2e @critical

Cenários de uso:

🚀 Deploy rápido

@smoke
Testes essenciais (5-10 min)
Validação básica de funcionalidades críticas

🔄 Regressão completa

@regression
Suite completa (30-60 min)
Execução noturna ou em releases

⚡ Testes de API

@api
Validação de contratos (10-15 min)
Execução a cada push

🎯 Funcionalidade específica

@login, @payment
Testes focados por feature
Debug e desenvolvimento

  • Feedback rápido para desenvolvedores (smoke em 5 min)
  • Otimização de recursos de CI/CD
  • Execução manual de workflows específicos no GitHub
  • Paralelização inteligente de testes
  • Estratégia de testes por ambiente (dev, staging, prod)
Resultado: Ciclo de feedback otimizado - do commit ao deploy em minutos, não horas.

Publicação de relatórios

Três Actions principais:

📤 upload-artifact

Salva o relatório

📥 download-artifact

Recupera

🌐 deploy-pages

Publica no GitHub Pages

Resultado: relatórios públicos e acessíveis.
Exemplo: https://joprestes.github.io/github_actions/

Boas práticas e depuração

Erros comuns:

❌ Branch gh-pages ausente
❌ Token inválido
❌ Caminhos incorretos
💡 Dica: use if: always() para garantir publicação mesmo em falhas.
Objetivo: manter o fluxo estável e rastreável.

Módulo 3

SonarQube — Qualidade Contínua Além dos Testes

O SonarQube é o olhar técnico sobre a saúde do código.
Complementa os testes ao medir o que eles não veem: estrutura, segurança e manutenibilidade.

O que é análise estática

Analisa código sem executá-lo, identificando problemas estruturais e de qualidade.

Categorias de problemas identificados:

🐛 Bugs

Problemas funcionais
• Null pointer exceptions
• Loops infinitos
• Condições sempre falsas
• Recursos não liberados

🔒 Vulnerabilidades

Riscos de segurança
• SQL Injection
• Cross-Site Scripting (XSS)
• Senhas hardcoded
• Validação insuficiente

💨 Code Smells

Problemas de manutenibilidade
• Métodos muito longos
• Código duplicado
• Complexidade ciclomática alta
• Nomes não descritivos

📊 Cobertura de testes

Métricas de qualidade
• Linhas cobertas por testes
• Branches testadas
• Funções sem cobertura
• Tendências ao longo do tempo

Benefícios para o QA:

  • Detecção precoce de problemas antes dos testes manuais
  • Priorização de áreas de risco para foco nos testes
  • Métricas objetivas para discussão com desenvolvedores
  • Histórico de evolução da qualidade do código
Analogia: O SonarQube é o check-up preventivo do código.
Identifica problemas antes que se tornem bugs em produção.

Quality Gate e métricas

Quality Gate = o "semáforo" da qualidade.
O pipeline só continua se as métricas forem aprovadas.

Critérios comuns:

  • Cobertura ≥ 80%
  • Zero bugs críticos
  • Duplicações < 5%
Resultado: deploy seguro e sustentável.

Interpretação e ação

O QA interpreta os relatórios e comunica riscos ao time.
